# YYC³ 团队通用 - AI协同开发文档

## 文档定位

**核心宗旨**: 为 AI 协同开发提供标准化、可追溯、可衔接的工作流程框架，确保「五维驱动」有效落地「五高五标五化」。

**适用场景**:

- AI 导师首次接触新项目的标准入职流程
- 跨会话、跨时间点的开发上下文无缝衔接
- 开发过程的全生命周期管理与文档沉淀

**核心理念**:

> **上下文存档 · 规划衔接 · 总结同步 · 随时可启**

---

## 一、AI导师首次接触项目 - 标准入职流程

### 1.1 原则：以底层代码为基，形成真实全局认知

```
┌─────────────────────────────────────────────────────────────┐
│                  AI 导师入职标准流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Phase 1: 环境感知 (必做)                                    │
│  ├─ 读取《系统上下文文档.md》 → 获取系统全景                  │
│  ├─ 读取《生产环境实际运行状态.md》 → 了解当前状态            │
│  ├─ 读取 README.md / CHANGELOG.md → 项目入口                │
│  └─ 扫描 package.json / requirements.txt → 依赖清单          │
│                                                              │
│  Phase 2: 代码审计 (必做)                                    │
│  ├─ 分析 src/ 目录结构 → 模块边界                            │
│  ├─ 审查核心配置文件 → config/, .env*                        │
│  ├─ 梳理数据库 schema / models → 数据模型                    │
│  └─ 检查测试覆盖情况 → 质量基线                              │
│                                                              │
│  Phase 3: 工作空间初始化 (必做)                              │
│  └─ 在 docs/ 创建本次会话工作目录                            │
│      格式: {项目名}-{导师标识}-{YYYYMMDD}                     │
│      示例: 0379-world-glm5-turbo-20260503                   │
│                                                              │
│  Phase 4: 输出审核分析文档 (必做)                            │
│  └─ 生成《项目现状审核报告.md》                               │
│      包含: 架构分析、问题发现、风险评估、改进建议              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 绝对禁止行为（红线）

| 行为                   | 原因                       | 后果             |
| ---------------------- | -------------------------- | ---------------- |
| ❌ 不读代码直接给方案   | 脱离实际，方案不可落地     | 视为无效输出     |
| ❌ 盲猜配置项值         | 可能引入安全风险或功能异常 | 必须标注"待确认" |
| ❌ 盲测未验证的命令     | 可能破坏生产环境           | 先在测试环境验证 |
| ❌ 跳过现状分析直接开发 | 违背"以代码为基"原则       | 退回重做         |
| ❌ 不保存文档就结束会话 | 上下文断裂，无法衔接       | 视为未完成       |

### 1.3 推荐行为（加分项）

| 行为                       | 说明                     |
| -------------------------- | ------------------------ |
| ✅ 对每个配置项都认真对待   | 配置是系统的DNA          |
| ✅ 准确理解整个系统架构     | 架构决定上限             |
| ✅ 提供专业建议而非随意操作 | 专业性体现价值           |
| ✅ 保护敏感信息             | 使用环境变量，不记录密钥 |
| ✅ 只在必要时使用敏感信息   | 最小权限原则             |
| ✅ 建议改进安全措施         | 主动防御优于被动修复     |

## 三、执行流程 - 规划驱动开发

### 3.1 标准开发循环

```
┌──────────────────────────────────────────────────────────────┐
│                    YYC³ 开发循环 (PDCA+)                      │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│   Plan (规划)                                                 │
│   ├─ 明确任务目标和验收标准                                     │
│   ├─ 分析影响范围和依赖关系                                     │
│   ├─ 制定实施方案和技术选型                                     │
│   └─ 更新 01-任务规划文档                                      │
│                         ↓                                     │
│   Do (执行)                                                   │
│   ├─ 按计划逐步实施                                            │
│   ├─ 实时记录到 02-执行日志                                    │
│   ├─ 关键操作前先确认                                          │
│   └─ 遇到阻塞立即上报                                         │
│                         ↓                                     │
│   Check (检查)                                                │
│   ├─ 对照验收标准自检                                          │
│   ├─ 运行 lint / typecheck / test                             │
│   ├─ 验证功能正确性和性能                                       │
│   └─ 记录检查结果                                             │
│                         ↓                                     │
│   Act (处理)                                                  │
│   ├─ 通过 → 标记完成，进入下一任务                              │
│   ├─ 不通过 → 分析原因，制定修复方案                             │
│   └─ 更新规划和进度                                           │
│                         ↓                                     │
│   Archive (归档) ← YYC³ 增强                                   │
│   ├─ 同步更新 02-执行日志                                      │
│   ├─ 生成 03-总结文档                                         │
│   ├─ 更新项目级 CHANGELOG                                      │
│   └─ 为下次会话留下完整上下文                                   │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

## 五、上下文衔接协议 - 跨会话 continuity

### 5.1 新会话启动标准流程

```

Step 1: 定位最新工作目录
        ↓
Step 2: 阅读 03-总结文档（上次会话的结尾）
        ↓
Step 3: 阅读 02-执行日志（了解最后在做什么）
        ↓
Step 4: 检查 01-任务规划（确认哪些已完成）
        ↓
Step 5: 验证当前代码状态（git status, git log）
        ↓
Step 6: 创建新的会话目录或续用现有目录
        ↓
Step 7: 向用户确认："我已了解上次进度，从 XXX 继续？"

```

### 5.2 上下文传递格式

当需要在文档中引用其他会话或外部上下文时，使用统一格式：

```markdown
<!-- CONTEXT_LINK:start -->
{
  "source": "{来源会话目录名}",
  "type": "decision/problem/solution/config",
  "summary": "{一句话摘要}",
  "location": "{具体文件路径}",
  "relevance": "{为什么这个上下文重要}"
}
<!-- CONTEXT_LINK:end -->
```

### 5.3 断点续传协议

如果会话被意外中断，按以下优先级恢复：

1. **查找最新的 03-总结文档**
2. **查看 02-执行日志的最后几条记录**
3. **运行 git diff 查看未提交的更改**
4. **对照 01-任务规划确认进度**

## 六、质量保障机制

### 6.1 代码质量门禁

每次代码变更必须通过以下检查：

| 检查项   | 命令/工具                               | 通过标准                  |

### B. 常见问题 Q&A

**Q: 每次都要写这么多文档会不会很慢？**
A: 前期投入是为了后期加速。一旦形成习惯，模板化的文档只需填充内容，反而减少重复解释的时间。

**Q: 如果会话很短（比如只改一行代码）也要写这么多吗？**
A: 可以简化。短会话只需更新执行日志并在总结中一句话带过。关键是"有记录"，而不是"形式主义"。

**Q: 敏感信息如何在文档中处理？**
A: 使用占位符格式：`${ENV_VAR_NAME}` 或 `[REDACTED]`。绝不记录真实的密钥、密码、Token。

**Q: 多个 AI 导师协作怎么办？**
A: 每个导师使用独立的会话目录，通过总结文档进行衔接。冲突解决以代码仓库的实际状态为准。
