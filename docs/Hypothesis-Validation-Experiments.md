## 一、实验框架概述

### 1.1 三大核心假设回顾

| 假设 | 原文 | 风险等级 | 验证时机 |
|------|------|---------|---------|
| **H1** | 量化交易者需要一站式平台（而非多工具拼接） | 中 | M1 后 (Week 5+) |
| **H2** | 量子计算能为量化策略优化带来可量化的优势 | 高 | M3 后 (Week 15+) |
| **H3** | AI + 量子 + 大数据三引擎协同能创造独特价值 | 高 | M4 后 (Week 19+) |

### 1.2 验证方法论

每个假设采用 **三级验证法**：

```
Level 1: 定性验证 (Qualitative)
  → 用户访谈 + 问卷 + 专家评审
  → 目标: 确认方向是否正确
  → 所需: 5-10 名目标用户

Level 2: 行为验证 (Behavioral)
  → 原型测试 + 使用数据埋点 + A/B 测试
  → 目标: 确认用户是否真的"用"
  → 所需: 20-50 名测试用户

Level 3: 量化验证 (Quantitative)
  → 统计显著性测试 + 性能基准对比
  → 目标: 确认优势的量化幅度
  → 所需: 100+ 次实验运行 / 50+ 名用户
```

### 1.3 实验道德约束

- 不采集用户 PII（个人身份信息）
- 所有实验使用模拟/历史数据，不涉及用户真实资金
- 量子算法对比使用相同输入数据集，确保公平性
- 用户测试征得知情同意

## 二、H1: 一站式平台需求验证

> **假设**: 量化交易者需要一站式平台（而非多工具拼接）

### 2.1 失败条件 (Falsification Criteria)

**H1 被证伪的条件**:
- > 60% 的目标用户表示更偏好多工具组合工作流
- 用户在 YYC 跨模块导航的频率 < 1 次/会话
- 用户在 YYC 中仅使用 ≤ 2 个模块（其余用外部工具）

### 2.2 实验 H1-E1: 工作流痛点访谈 (Level 1)

**目标**: 验证多工具切换确实是用户痛点

| 参数 | 值 |
|------|----|
| 方法 | 半结构化深度访谈 |
| 样本 | 8-12 名量化交易从业者（含散户 4 + 小机构 4 + 机构 2-4） |
| 时长 | 每人 30-45 分钟 |
| 渠道 | 线上视频 / 量化社区招募 |

**访谈提纲**:

```
第一部分: 当前工作流 (10 min)
├── Q1: 请描述你一天的典型量化交易工作流程
├── Q2: 你目前使用哪些工具？（列举工具名 + 用途）
├── Q3: 这些工具之间如何传递数据？手动还是自动？
└── Q4: 在工具切换过程中，你遇到过哪些具体的效率问题？

第二部分: 一站式需求探测 (10 min)
├── Q5: 如果有一个平台能覆盖你 80% 的工作流，你会考虑使用吗？
├── Q6: 你最希望哪些功能整合在同一平台中？（排序 Top 5）
├── Q7: 你认为一站式平台的最大风险是什么？
└── Q8: 你愿意为一站式平台付多少月费？（开放定价探测）

第三部分: YYC 原型展示 (15 min)
├── Q9: [展示 8 模块] 这些模块的覆盖度如何？是否缺少关键功能？
├── Q10: [展示 CrossModuleBar] 跨模块导航体验如何？
├── Q11: 你最先会尝试哪个模块？为什么？
└── Q12: 整体印象打分 (1-10) + 改进建议
```

**成功标准**:
- ≥ 70% 受访者确认多工具切换是显著痛点（Severity ≥ 7/10）
- ≥ 60% 受访者对 YYC 一站式方案表达强烈兴趣（Interest ≥ 8/10）
- 平均模块覆盖度评分 ≥ 3.5/5

**数据记录模板**:

| 受访者 | 角色 | 当前工具数 | 切换痛点 (1-10) | 一站式兴趣 (1-10) | 模块覆盖度 (1-5) | 首选模块 | 月付意愿 |
|--------|------|-----------|----------------|-------------------|-----------------|---------|---------|
| P01 | 加密散户 | | | | | | |
| P02 | 私募量化 | | | | | | |
| ... | ... | | | | | | |

### 2.3 实验 H1-E2: 模块使用热力图 (Level 2)

**目标**: 验证用户是否真的使用多个模块

| 参数 | 值 |
|------|----|
| 方法 | 前端埋点 + 使用分析 |
| 样本 | 20-50 名内测用户，使用 ≥ 7 天 |
| 指标 | 模块切换频率、会话时长分布、跨模块操作链 |

**埋点方案** (轻量化，不采集 PII):

```typescript
// 在 App.tsx 的 handleModuleChange 中埋点
interface UsageEvent {
  eventType: 'module_switch' | 'sub_switch' | 'cross_module_nav' | 'action';
  from: string;        // 来源模块/子页面
  to: string;          // 目标模块/子页面
  timestamp: number;   // Unix ms
  sessionId: string;   // 随机会话 ID (非 PII)
  duration: number;    // 在前一页面停留时长 (ms)
}

// 存储: localStorage 本地聚合 → 用户手动导出 JSON
// 或: Supabase anonymous analytics table (Phase 5)
```

**分析维度**:

```
1. 模块热力图: 每个模块的访问次数占比
   → 预期: 至少 4 个模块的访问占比 > 5%
   → 风险信号: 1-2 个模块占 > 80% 访问

2. 跨模块转化路径:
   → 核心路径: market → strategy → risk → trade (MSRT 链)
   → 预期: ≥ 30% 会话包含 ≥ 3 个不同模块
   → 风险信号: < 15% 会话包含 3+ 模块

3. CrossModuleBar 使用率:
   → 预期: ≥ 20% 的跨模块导航通过 CrossModuleBar 触发
   → 风险信号: < 5% (说明该功能不被感知)

4. 会话深度:
   → 预期平均会话时长: ≥ 15 分钟
   → 预期平均模块切换次数: ≥ 4 次/会话
```

**成功标准**:
- ≥ 4 个模块的访问占比 > 5% (模块使用分散度)
- ≥ 30% 的会话包含 3+ 不同模块 (多模块使用)
- 平均会话模块切换 ≥ 4 次 (频繁跨模块)

### 2.4 实验 H1-E3: 工具切换效率 A/B 对比 (Level 3)

**目标**: 量化一站式 vs 多工具工作流的效率差异

| 参数 | 值 |
|------|----|
| 方法 | 受控任务实验 |
| 样本 | 20 名量化交易者（随机分 A/B 组各 10 人） |
| 任务 | 标准化量化工作流任务（见下） |

**标准化任务**:

```
任务: "发现 BTC 价格异常 → 分析原因 → 调整策略参数 → 评估风险 → 模拟下单"

A 组 (YYC 一站式):
  使用 YYC-QATS 平台完成全流程

B 组 (多工具组合):
  使用 TradingView (行情) + QuantConnect/聚宽 (策略) + Excel (风控) + 券商终端 (交易)
  
测量指标:
  - 任务完成时间 (秒)
  - 数据手动搬运次数
  - 任务完成正确率
  - 主观满意度 (NASA-TLX 量表)
  - 错误次数 (误操作/数据错误)
```

**成功标准**:
- A 组任务完成时间比 B 组快 ≥ 25% (p < 0.05)
- A 组数据搬运次数比 B 组少 ≥ 60%
- A 组 NASA-TLX 总负荷评分比 B 组低 ≥ 15%

## 三、H2: 量子计算可量化优势验证

> **假设**: 量子计算能为量化策略优化带来可量化的优势

### 3.1 失败条件 (Falsification Criteria)

**H2 被证伪的条件**:
- 量子算法在策略优化任务上的结果不优于经典算法 (p > 0.05)
- 量子算法的计算耗时在实际问题规模下 > 经典算法的 2x
- 量子噪声导致结果不可靠（结果方差 > 经典方法的 3x）

### 3.2 实验 H2-E1: 专家感知价值访谈 (Level 1)

**目标**: 确认量子计算在金融量化领域的感知价值

| 参数 | 值 |
|------|----|
| 方法 | 专家深度访谈 |
| 样本 | 6-8 名（量化研究员 3 + 量子计算研究者 2 + 跨领域 1-3） |
| 时长 | 每人 45-60 分钟 |

**访谈提纲**:

```
第一部分: 量子认知 (15 min)
├── Q1: 您认为量子计算在金融领域最有前景的应用场景是什么？
├── Q2: 您认为量子优势 (Quantum Advantage) 在金融领域何时可能实现？
└── Q3: 当前量子计算在金融应用中面临的最大障碍是什么？

第二部分: 场景评估 (20 min)
├── Q4: [展示量子模块] 这些应用场景的优先级如何排序？
├── Q5: QAOA 组合优化——您认为在多少资产规模时量子有实际优势？
├── Q6: 量子蒙特卡洛 VaR——相比经典 MC，您预期的加速比是多少？
└── Q7: 量子机器学习——您认为 QML 在金融预测中是否有独特优势？

第三部分: 产品化评估 (15 min)
├── Q8: 将量子算法"封装为开箱即用的金融组件"，这个产品策略您如何评价？
├── Q9: 量化从业者是否需要理解量子原理才能使用这些功能？
└── Q10: 您会推荐量化团队现在就开始储备量子能力吗？(1-10)
```

**成功标准**:
- ≥ 75% 专家认为量子金融有实际应用前景 (2-5 年内)
- ≥ 60% 专家对"封装为业务组件"策略评价正面 (≥ 7/10)
- 至少 2 个应用场景被 > 50% 专家评为"高优先级"

### 3.3 实验 H2-E2: 量子 vs 经典基准测试 (Level 3)

**目标**: 在标准化金融任务上量化量子算法的性能

| 参数 | 值 |
|------|----|
| 方法 | 受控基准测试 |
| 算法对 | QAOA vs 经典优化 / QMC vs 经典MC / VQE vs 经典特征值 |
| 运行次数 | 每个任务 100 次独立运行 |
| 硬件 | 量子模拟器 (浏览器端) + 可选: IBM Qiskit Runtime |

**基准测试矩阵**:

```
┌──────────────────────────────────────────────────────────────────┐
│  测试 1: 组合优化 (QAOA vs 经典)                                  │
│  ├── 问题: 在 N 个资产中选择最优权重，最大化夏普比率               │
│  ├── 规模梯度: N = 5, 10, 15, 20, 30, 50 个资产                  │
│  ├── 经典对照: 均值-方差优化 (Markowitz) + 遗传算法               │
│  ├── 量子方法: QAOA (p=1,2,4 层) + 量子退火                     │
│  ├── 指标: 最优解质量 / 计算耗时 / 解的稳定性                    │
│  └── 预期: N > 20 时量子开始显现优势                              │
├──────────────────────────────────────────────────────────────────┤
│  测试 2: 风险计算 (QMC vs 经典 MC)                               │
│  ├── 问题: 计算组合 VaR (95%/99%) — 多资产蒙特卡洛              │
│  ├── 精度梯度: 10K / 100K / 1M / 10M 路径                       │
│  ├── 经典对照: 标准蒙特卡洛 + 对偶变量法 + 重要性抽样           │
│  ├── 量子方法: 量子振幅估计 (QAE) + Grover 加速                 │
│  ├── 指标: VaR 估计精度 / 收敛速度 / 置信区间宽度               │
│  └── 预期: 量子在高精度需求时有 quadratic speedup               │
├──────────────────────────────────────────────────────────────────┤
│  测试 3: 因子分析 (VQE vs 经典)                                  │
│  ├── 问题: 提取资产收益率协方差矩阵的主成分                      │
│  ├── 矩阵规模: 10x10, 20x20, 50x50                              │
│  ├── 经典对照: NumPy SVD / Lanczos 迭代                         │
│  ├── 量子方法: VQE + HHL 量子线性系统                            │
│  ├── 指标: 特征值精度 / 计算耗时 / 量子比特数需求               │
│  └── 预期: 当前量子硬件下可能无优势，但模拟器验证可行性         │
└──────────────────────────────────────────────────────────────────┘
```

**数据记录模板**:

| 测试ID | 算法 | 问题规模 | 解质量 | 耗时(ms) | 稳定性(std) | 量子比特数 | 备注 |
|--------|------|---------|--------|---------|------------|-----------|------|
| T1-Q-05 | QAOA-p2 | N=5 | | | | | |
| T1-C-05 | Markowitz | N=5 | | | | N/A | |
| ... | ... | ... | | | | | |

**成功标准**:
- **最低门槛**: 至少 1 个测试场景中量子算法在解质量上不劣于经典 (非劣效性 p < 0.05)
- **目标门槛**: 至少 1 个测试场景中量子算法在解质量上显著优于经典 (优效性 p < 0.05)
- **理想门槛**: 在 N ≥ 20 的组合优化中量子解质量优于经典 ≥ 5%

### 3.4 实验 H2-E3: 用户感知量子价值 A/B (Level 2)

**目标**: 验证用户是否感知到量子功能的额外价值

| 参数 | 值 |
|------|----|
| 方法 | A/B 功能测试 |
| 样本 | 30 名内测用户 (A/B 各 15) |
| 时长 | ≥ 14 天使用期 |

```
A 组: 完整 YYC (含量子模块)
  → 策略优化提供 "经典优化" + "量子优化" 双引擎
  → 风控模块提供 "经典VaR" + "量子VaR" 对比

B 组: YYC 去量子版 (量子模块标记为"即将推出")
  → 策略优化仅提供 "经典优化"
  → 风控模块仅提供 "经典VaR"

测量:

  - 用户对平台专业度的评分差异 (A vs B)
  - NPS (Net Promoter Score) 差异
  - "是否愿意为量子功能额外付费"的比例差异
```

**成功标准**:
- A 组 NPS 比 B 组高 ≥ 10 分
- A 组中 ≥ 40% 用户至少使用过 1 次量子功能
- A 组中 ≥ 25% 用户表示愿意为量子功能额外付费

## 四、H3: 三引擎协同价值验证

> **假设**: AI + 量子 + 大数据三引擎协同能创造独特价值

### 4.1 失败条件 (Falsification Criteria)

**H3 被证伪的条件**:
- 三引擎协同工作流的完成率 < 单引擎工作流的 80%
- 协同输出 (如量子加速 AI 模型训练) 的质量不优于独立运行
- 用户不主动使用跨引擎功能，或使用后评价为"增加复杂度无实际收益"

### 4.2 实验 H3-E1: 协同工作流原型测试 (Level 2)

**目标**: 验证用户能否顺畅完成三引擎协同工作流

| 参数 | 值 |
|------|----|
| 方法 | 任务导向原型测试 |
| 样本 | 15 名内测用户 |
| 任务 | 3 个标准化三引擎协同任务（见下） |

**三个标准化协同任务**:

```
任务 A: "数据驱动策略优化"
  大数据 → AI → 策略
  流程: 在 BigData 中选择数据源 → 在 Model 中训练预测模型 
        → 在 Strategy 中使用模型信号生成策略
  测量: 任务完成率 / 步骤完成时间 / 用户满意度

任务 B: "量子增强风控"
  量子 → 风控 → 交易
  流程: 在 Quantum 中运行 QAOA 组合优化 → 在 Risk 中查看量子VaR
        → 在 Trade 中根据风控结果调整持仓
  测量: 任务完成率 / 跨模块导航流畅度 / 决策信心评分

任务 C: "全链路闭环"
  大数据 → AI → 量子 → 策略 → 风控 → 交易
  流程: 数据采集 → 模型训练 → 量子优化参数 → 策略回测 
        → 风险评估 → 模拟交易
  测量: 端到端完成率 / 总耗时 / 放弃节点分析
```

**数据采集方式**:

```
1. 屏幕录制 (用户授权):
   → 记录操作路径、犹豫点、错误操作

2. 出声思考法 (Think Aloud):
   → 用户边操作边说出想法
   → 记录困惑点、惊喜点

3. 任务后问卷:
   → SUS (System Usability Scale) 评分
   → 协同价值感知评分 (1-10)
   → 开放式改进建议

4. 跨模块转化漏斗:
   → 记录每个步骤的完成 / 放弃 / 回退
   → 找出流失率最高的环节
```

**成功标准**:
- 任务 A/B 完成率 ≥ 70%
- 任务 C (全链路) 完成率 ≥ 50%
- SUS 评分 ≥ 68 (可用性基准线)
- 协同价值感知 ≥ 6.5/10

---

### 4.3 实验 H3-E2: 协同 vs 独立性能对比 (Level 3)

**目标**: 量化三引擎协同的实际效果增量

| 参数 | 值 |
|------|----|
| 方法 | 受控性能实验 |
| 配置 | 4 种引擎组合 × 3 个任务 × 100 次运行 |

**实验矩阵**:

```
配置 1: 仅经典 (无AI/无量子)
  → 经典统计因子 + Markowitz优化 + 标准MC VaR

配置 2: 经典 + AI (无量子)
  → AI因子 + AI参数优化 + 标准MC VaR

配置 3: 经典 + 量子 (无AI)
  → 经典因子 + QAOA优化 + 量子MC VaR

配置 4: AI + 量子 + 大数据 (三引擎协同)
  → AI因子(大数据驱动) + QAOA优化(AI初始化) + 量子MC VaR

任务 1: 20 资产组合优化
  → 指标: 夏普比率 / 最大回撤 / 年化收益
  
任务 2: 策略参数搜索 (5维参数空间)
  → 指标: 最优参数找到时间 / 参数稳健性
  
任务 3: 组合 VaR 计算 (10 资产/1M路径)
  → 指标: VaR 精度 / 计算耗时 / 置信区间
```

**数据记录模板**:

| 任务 | 配置 | 运行# | 夏普 | 最大回撤 | 年化收益 | 耗时(ms) | 备注 |
|------|------|-------|------|---------|---------|---------|------|
| T1 | Config1 | 001 | | | | | |
| T1 | Config4 | 001 | | | | | |
| ... | ... | ... | | | | | |

**统计检验方案**:
- 组间比较: Kruskal-Wallis 非参数检验 (4 组)
- 事后比较: Dunn's test with Bonferroni correction
- 效应量: Cohen's d (衡量实际差异大小)

**成功标准**:
- **最低门槛**: Config4 在至少 1 个任务上显著优于 Config1 (p < 0.05, d > 0.3)
- **目标门槛**: Config4 在至少 2 个任务上显著优于 Config1 和 Config2 (p < 0.01)
- **理想门槛**: Config4 在所有 3 个任务上均为最优配置 (夏普提升 ≥ 8%)

### 4.4 实验 H3-E3: 工作流完整度付费意愿 (Level 1)

**目标**: 验证三引擎协同是否影响用户付费意愿

| 参数 | 值 |
|------|----|
| 方法 | 联合分析 (Conjoint Analysis) |
| 样本 | 40-60 名目标用户 (线上问卷) |

**联合分析因素**:

```
因素 1: 市场数据覆盖度
  Level A: 加密货币 (6 币种)
  Level B: 加密+美股 (20 资产)
  Level C: 全品种 (加密+美股+A股+期货+外汇, 100+ 资产)

因素 2: AI 能力
  Level A: 无 AI
  Level B: AI 预测信号
  Level C: AI 模型训练 + 部署 + 监控

因素 3: 量子计算
  Level A: 无量子
  Level B: 量子模拟器 (教育/演示)
  Level C: 量子云服务 (真实量子硬件)

因素 4: 月费
  Level A: $29/月
  Level B: $99/月
  Level C: $299/月
  Level D: $999/月

→ 正交设计: 生成 16 个产品组合方案
→ 用户对每个方案评分 (1-10 购买意愿)
→ 分析: 各因素的 part-worth utility
```

**成功标准**:
- 量子功能的 part-worth utility > 0 (正向价值贡献)
- AI+量子组合的交互效应为正 (协同 > 简单相加)
- 三引擎完整版 (C+C+C) 的平均支付意愿 ≥ $99/月

## 六、数据采集基础设施

### 6.1 前端埋点模块设计

```typescript
// /src/app/services/AnalyticsService.ts

interface AnalyticsEvent {
  type: 'module_view' | 'sub_view' | 'feature_use' | 'cross_nav' | 'error';
  payload: Record<string, string | number>;
  timestamp: number;
  sessionId: string;
}

class AnalyticsService {
  private events: AnalyticsEvent[] = [];
  private sessionId: string;
  
  constructor() {
    this.sessionId = crypto.randomUUID();
  }

  track(type: AnalyticsEvent['type'], payload: AnalyticsEvent['payload']): void {
    this.events.push({ type, payload, timestamp: Date.now(), sessionId: this.sessionId });
    // 本地存储，不外传
    this.persist();
  }

  private persist(): void {
    try {
      const key = `yyc_analytics_${new Date().toISOString().slice(0,10)}`;
      localStorage.setItem(key, JSON.stringify(this.events));
    } catch { /* storage full - rotate */ }
  }

  export(): string {
    return JSON.stringify(this.events, null, 2);
  }
}
```

## 七、决策矩阵与 Pivot 条件

### 7.1 假设验证决策树

```
H1 验证结果:
├── ✅ 强支持 (所有标准达成)
│   → 加大一站式整合投入，8 模块全推进
│
├── ⚠️  弱支持 (部分标准达成)
│   → 聚焦高使用率的 4-5 个核心模块
│   → 低使用率模块降优先级或合并
│
└── ❌ 不支持 (多数标准未达成)
│   → Pivot: 从一站式转为模块化 SaaS
│   → 允许用户按需购买单模块
│   → 保留跨模块数据互通作为增值

H2 验证结果:
├── ✅ 强支持 (量子有可量化优势)
│   → 量子作为核心差异化，加大投入
│   → 商业定价中量子功能作为高级版核心
│
├── ⚠️  弱支持 (量子有潜力但当前无明显优势)
│   → 量子定位为"前沿研究实验室"
│   → 作为品牌差异化 (非核心功能)
│   → 持续跟踪量子硬件进展
│
└── ❌ 不支持 (量子无优势)
    → 降低量子投入到最低维护
    → 将资源转移到 AI 和数据管线
    → 保留量子模块 UI 作为未来入口

H3 验证结果:
├── ✅ 强支持 (协同 > 独立)
│   → 深化三引擎集成，强调协同工作流
│   → 商业叙事: "三引擎协同是核心壁垒"
│
├── ⚠️  弱支持 (部分协同有效)
│   → 聚焦有效的协同对 (如 AI+数据)
│   → 降低无效协同的复杂度
│
└── ❌ 不支持 (协同无额外价值)
    → 允许引擎独立使用
    → 简化跨引擎工作流
    → 重新定位: "模块化量化工具箱"
```
