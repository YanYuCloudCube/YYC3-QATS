#### 第一部分: 当前工作流 (15 min)

**Q1: 日常工作流** (开放式)
> "请描述一下您一天的典型量化交易工作流程。从早上打开电脑开始，到收盘后，您都会做什么？"

- 追问: "这个流程中哪个环节耗时最长？"

**Q2: 工具清单** (列举式)

## 五、分析框架

### 5.1 定性编码方案

对所有访谈文本进行以下维度编码:

```
痛点编码:
  PAIN-SWITCH    工具切换耗时/上下文丢失
  PAIN-DATA      数据格式不一致/手动搬运
  PAIN-SYNC      多工具状态不同步
  PAIN-COST      工具费用过高
  PAIN-LEARN     学习曲线陡峭
  PAIN-PERF      性能/延迟问题
  PAIN-OTHER     其他痛点

需求编码:
  NEED-REALTIME  实时数据需求
  NEED-BACKTEST  回测需求
  NEED-RISK      风控需求
  NEED-AI        AI/ML 需求
  NEED-QUANTUM   量子计算需求
  NEED-ALLONE    一站式整合需求
  NEED-MOBILE    移动端需求
  NEED-OTHER     其他需求

评价编码:
  EVAL-POS       正面评价
  EVAL-NEG       负面评价
  EVAL-SUG       改进建议
  EVAL-WOW       "Wow" 时刻
  EVAL-CONFUSE   困惑/不理解
```

### 5.2 关键指标计算

```
痛点确认率 = (痛点严重度 >= 7 的人数) / 总受访人数
  目标: >= 70%

一站式兴趣率 = (整体评分 >= 8 的人数) / 总受访人数
  目标: >= 60%

平均模块覆盖度 = SUM(覆盖度评分) / 总受访人数
  目标: >= 3.5

功能优先级共识 = 排名前 3 中出现频率最高的功能
  输出: Top 3 最需整合功能

定价锚点 = MEDIAN(可接受月费)
  输出: 定价策略参考
```

### 5.3 决策规则

```
IF 痛点确认率 >= 70% AND 一站式兴趣率 >= 60% AND 覆盖度 >= 3.5:
  → H1 Level 1 通过 → 进入 H1-E2 行为验证
  → 记录功能优先级，指导 Phase 2 开发优先级

IF 痛点确认率 >= 70% BUT 一站式兴趣率 < 60%:
  → 方向正确但方案需调整
  → 深入分析抗拒原因（安全顾虑？功能深度不足？）
  → 追加 4 人补充访谈，聚焦抗拒点

IF 痛点确认率 < 70%:
  → H1 可能需要修正
  → 分析: 是样本偏差还是真实不成立？
  → 决策: 扩大样本 or 调整假设定义

IF 覆盖度 < 3.5:
  → 模块设计需要重大调整
  → 分析缺失功能集中在哪些领域
  → 输出: 模块增减建议
```

## 七、风险与缓解

| 风险 | 概率 | 缓解 |
|------|------|------|
| 招募不足 8 人 | 中 | 扩大渠道 (Twitter/Reddit/V2EX)；提高礼品金额 |
| 受访者不说真话 (社会期望偏差) | 中 | 使用"假设第三方"话术；先问痛点再展示产品 |
| 量子功能超出受访者认知 | 高 | 用类比解释 ("量子是一种新的高速计算方式，可以更快找到最优策略参数") |
| 访谈中受访者对产品太礼貌/不批评 | 中 | 明确鼓励批评 ("您的批评比赞美更有价值") |
| 样本偏向技术型用户 | 中 | 刻意招募非技术背景的交易者 (如纯策略型/直觉型) |
